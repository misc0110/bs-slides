Memory Management

Address Space Layout in SWEB

  • Every process has its own Address Space
  • 0 - 2 GiB Program
  • 2 - 3 GiB Kernel (the same for all processes)
  • 3 - 4 GiB Identity-Mapping

Page Directories

  • Every process (and Kernel Thread) has its own Page Directory
  • It's needed to map the Virtual Address Space (4GB) to the Physical Address Space
  • A system call jumps into the Kernel Space, but the Page Directory stays the same
  • The Identity Mapping is used for controlled access to the Physical Address Space

Function Call

Function Calls in Detail (cdecl)

  • Caller
    • Pushes arguments onto the stack (push or mov)
    • Calls the function (call)
  • CPU pushes the return address onto the stack
  • The control flow is given to the function (EIP-register is manipulated)
  • Function
    • Executes the code
    • Sets the EAX-register to the return value
    • Executes the ret-instruction
  • CPU takes the return address from the stack
  • The control flow is given to the calling function again (change the EIP-register)
  • Caller cleans up the stack

C Program

int add(int a, int b)
{
  return a + b;   
}

int main() 
{
  int a = 7;
  int b = 14;
  int c = 0;
  
  c = add(a, b);
  
  return c;

Synchronization Mechanism

Process Synchronization

  • Problem:
    • Several processes and threads share the same resources
    • They can be destroyed by uncoordinated access
    • Resources may become invalid
    • → Security mechanisms required
  • Counter Actions
    • Identify sections with shared resources: "critical sections"
    • Protect access to the critical sections
  • Some Mechanisms
    • Mutex
    • Semaphore
    • Condition Variables

Mutex - Mutual Exclusion

  • Mutex - Mutual Exclusion
  • Every mutex has a locking variable (set by using an atomic operation, for instance xchg)
  • To access a critical section, every thread needs to hold a lock
  • The thread wants to access the section, and the lock is free → thread acquires the lock
  • Lock is not free → thread is going to sleep (until the lock is released)
  • Sleeping threads → sleepers list
  • Thread exits the critical section and releases the lock (!) → next sleeping thread is woken up

Semaphore

  • Semaphore: usually a signal post for trains
  • It has a counter, initialized with n
  • Thread wants to use the resource and the counter is bigger than 0 → decrement the counter by 1
  • The counter is 0 → thread is going to sleep (until the counter is bigger than 0)
  • Sleeping threads → sleepers list
  • Thread doesn't need the resource any longer and increments the counter by 1 (!) → next sleeping thread is woken up
  • Usage: producer/consumer (how many items are within the queue), numeric limited resources
  • Not yet implemented in SWEB!

Condition Variable

  • Used to synchronize processes/threads within a critical section
  • Citical sections are protected by a mutex
  • A thread can only continue within a critical section if a condition is fulfilled
  • The thread is going to sleep until a condition is fulfilled
  • Meanwhile, another thread can enter the critical section
  • When a condition is fulfilled, one or all threads waiting get informed
  • Every condition needs its own variable

Condition Variable con't

  • Wakeup- call - two reactions possible:
    • Mesa Style: The sleeping thread waits until the thread in the protected section has finished
    • Hoare Style: The sleeping thread starts immediately, and the running thread is put to sleep
  • Mesa style more common
  • Mutex and CV interact hand in hand
  • Wakeup-calls are not stored

Multithreading

  • How do our processes realize multithreading
  • Every process has its own virtual address space
  • Multiple executed threads per process (number only limited by hardware)
  • Managed by the kernel (easier than user space managed)

Multithreading Design Decisions

  • What happens when the first thread finishes?
  • When to clean-up user space and loader?
  • How to detect that a function running as a thread returns?
  • Where shall the stacks be placed?
  • What happens if the process calls fork/exec?

Multithreading in SWEB

  • One kernel thread per UserProcess at the moment
  • We want to have multiple threads (for instance UserThread)
  • ... which share the same page directory and the same loader
  • Protect shared ressources!!!
  • Provide test cases (the more the better)

Multithreading

  • POSIX Thread Library
  • man pthreads (or search for pthreads in the WWW)
  • Decide which syscalls are useful
  • At least the following functionality has to be implemented:
    • Create threads (pthread_create)
    • Exit threads (pthread_exit)
    • Kill threads (pthread_cancel)
    • Wait for threads (pthread_join)

System Call SC_FORK



  • pid_t fork(void);
  • create a duplicate of the calling process
  • Is needed to execute additional programs within the user space
  • Return Value:
    • 0 in the child process
    • The child's process-id in the parent process

Fork Design Decisions

  • What happens if the process has several threads?
  • What happens with the used ressources? (files, mutexes, ...)?
  • How to ensure that the copy procedure is ''atomic''?
  • Advanced: Is it possible that processes share pages (especially code)?

System Call SC_EXEC



  • int execv(const char *path, char *const argv[]);
  • replaces the image of the current process by an image given by the path
    • (take a look at the Loader) Loader
  • is used in combination with fork to start programs
  • Return value only relevant in case of an error

Example for fork/exec/waitpid

int child_status;

int pid = fork();

if(pid == 0) { // child process
    exec("programm.sweb");
} else if(pid > 0) { // parent process
    waitpid(pid, &child_status, WEXITED);
}

System Call SC_EXEC with parameters

  • int execv(const char *path, char *const argv[]);
  • argv should be handed over to the main-function of the new process
  • Where shall the parameters be placed at?
  • How to give them to main?

System Call SC_WAITPID

pid_t waitpid(pid_t pid, int *status, int options);

  • Wait until the child process with the given PID terminates
  • Needed for a working shell
  • The return value of the process is stored at the status address

System Call SC_SLEEP

  • unsigned sleep(unsigned seconds);
  • The calling thread is put to sleep for a specified time
  • The time is given in seconds
  • The time shall be as precise as possible (!)
  • "yield once" is not enough
  • Can be realized using time stamp-counter or real time clock

System Call SC_CLOCK

  • clock_t clock(void);
  • Returns the time a process is running on the CPU since it was started
  • Time in clocks (see man-page)
  • The time shall be as precise as possible (timestamp-counter or real time clock)
  • compare to SC_SLEEP

Resource Management

  • Every process has different resources available (file descriptors, mutexes, shared memory, ...)
  • These resources have to be managed (threads are only allowed to access valid resources)
  • These resources have to be protected (other processes shall not simply access/change them)
  • A good implementation base makes future tasks easier to implement

Filesystem Systemcalls

  • Syscalls: open, close, read, write still exist
  • File descriptors have to be allocated per process ("process resource")
  • An invalid access (does not exist/belongs to a different process) to a file descriptor has to be handled
  • FDs still exist
    • 0: stdout
    • 1: stdin
  • File system-access exists, "only" the management for the processes is missing

File System Systemcalls cont'd

  • It is recommended to create files and folders in /user_progs (no additional mounting required).
  • Interesting when testing: manual mount the partition (for instance inside the Linux terminal), sweb-bin-folder:
    mount SWEB-flat.vmdk -oloop,offset=10321920 <target-folder>
  • Or in newfs: ./utils/mount.sh
  • (Take care of the write access!)

System Call SC_PIPE

  • int pipe(int filedes[2]);
  • Generates two file descriptors
  • The data written to fildes[1] can be read out from fildes[0]
  • Used for shell I/O redirection (and several other things)
  • Related: How can processes share file descriptors (if they want to)? (!)

System Call SC_ATEXIT

  • int atexit(void (*func)());
  • The registered function func shall be executed when the process terminates
  • Call the function in reverse order when exit is called
  • At least 32 functions shall be registerable
  • Who executes the functions, after all threads have been terminated?

System Call SC_SHUTDOWN

  • Controlled shut-down of the system
  • All Processes have to be terminated before (do not forget atexit!)
  • Shut down the PC (ACPI)

Semaphore
